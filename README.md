# China Jobs — AI 对中国工作岗位的替代分析

灵感来源：[Karpathy/jobs](https://github.com/karpathy/jobs) — 对美国 342 个职业的 AI 替代可能性评分项目。

## 项目目标

对中国所有职业进行 AI 替代可能性评分（0-10），并通过交互式可视化呈现结果。

## Karpathy 项目的做法

1. 从美国劳工统计局（BLS）抓取全部 342 个职业分类及就业人数
2. 用 LLM + 详细评分标准（rubric）给每个职业打 0-10 分
3. 构建交互式 treemap：矩形面积 = 就业人数，颜色 = AI 暴露程度
4. 核心判断逻辑：如果工作产出是数字化的，且可以完全在家办公，AI 暴露程度天然就高

## 中国版可行性结论

**完全可行，且有独特价值。**

### 数据源对比

| 维度 | 美国版 | 中国版 |
|------|--------|--------|
| 职业分类 | BLS SOC（342 个职业） | 《中华人民共和国职业分类大典（2022年版）》（8 大类、79 中类、449 小类、1639 个细类） |
| 就业人数 | BLS Occupational Employment Statistics | 国家统计局年鉴 / 人社部就业统计 / 各行业报告 |
| LLM 评分 | OpenAI API | Claude API |
| 可视化 | D3.js treemap | D3.js / ECharts treemap |

### 中国版的独特价值

1. **职业粒度更细** — 1639 个细类 vs 美国的 342 个，分析更精细
2. **制造业维度** — 中国制造业就业占比远高于美国，需要额外考虑工业机器人/自动化替代（不仅仅是 AI）
3. **中美对比** — 同类职业在中美的评分差异本身就很有话题性（如基础设施、数字化程度差异）
4. **市场空白** — 中文互联网上目前没有类似的高质量开源项目
5. **传播潜力** — 就业话题在中国社交媒体上天然高关注度

### 需要适配中国国情的地方

- **评分标准（rubric）需要调整**：Karpathy 的 rubric 核心是"工作产出是否数字化 + 能否远程办公"，但中国很多岗位的数字化程度、工作方式与美国不同
- **制造业/体力劳动权重**：中国制造业、建筑业、农业就业人口占比更大，需要加入机器人/自动化替代的评估维度
- **就业数据获取难度**：中国的职业级就业数据不如美国 BLS 公开透明，可能需要从多个来源拼接，或使用行业级估算
- **政策因素**：部分岗位受政策保护（如公务员体系），AI 替代的实际速度与技术可行性可能脱节

### 技术管线

```
1. 数据采集
   ├── 爬取《职业分类大典》职业列表（人社部官网 / 公开 PDF）
   └── 收集就业人数数据（国家统计局年鉴 / 行业报告）

2. LLM 评分
   ├── 设计中国版评分 rubric
   ├── 用 Claude API 批量评分 1639 个职业
   └── 人工抽查校准

3. 可视化
   ├── 交互式 treemap（面积=就业人数，颜色=AI暴露度）
   ├── 支持按大类/中类/小类下钻
   └── 搜索 & 筛选功能

4. 部署
   └── GitHub Pages / Vercel 静态站
```

### 预估工作拆分

| 阶段 | 主要工作 |
|------|----------|
| 数据采集 | 职业列表结构化、就业数据收集与清洗 |
| Rubric 设计 | 针对中国国情设计评分标准，小批量测试迭代 |
| 批量评分 | Claude API 调用，结果缓存，异常处理 |
| 可视化 | treemap 开发，交互功能，移动端适配 |
| 发布 | 部署、README、社交媒体传播 |

### 主要风险

1. **就业人数数据缺失** — 细类级别的就业数据可能拿不到，需要降级到小类或中类
2. **LLM 评分一致性** — 1639 个职业批量打分，需要确保评分标准一致
3. **敏感性** — 涉及就业话题，传播时需注意措辞
